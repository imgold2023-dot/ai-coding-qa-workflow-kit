# AI 编程工具安全门槛清单

版本：0.1

## 这是什么

这是一份给独立开发者、QA 工程师和小型技术团队使用的实用清单。适用于你准备使用 AI 编程工具、GitHub repo、agent workflow、MCP server、浏览器自动化脚本或第三方模板之前。

在你安装、运行、连接或信任一个工具前，先用它过一遍。

这份清单不保证绝对安全。它的作用是帮你慢下来，避开明显风险，并判断一个工具是否适合先用假数据试跑。

## 快速结论

先用这张表判断：

| 结论 | 含义 |
| --- | --- |
| Use | 风险较低，可以用假数据做本地测试。 |
| Use with isolation | 有价值，但必须隔离测试，例如临时文件夹、容器、虚拟机或临时账号。 |
| Study only | 只借鉴思路，不安装、不连接。 |
| Reject | 不进入本项目。 |

## 硬性拒绝规则

只要满足任一条件，直接拒绝：

- 试跑就要求银行卡、收款账号、支付密码、真实 cookie、浏览器配置或生产 token。
- 要求你把密钥粘贴到第三方网页或不明脚本里。
- 没有 license，或 license 明显不允许你的目标用途。
- 安装阶段下载不明二进制文件。
- 存在无法解释的 `postinstall`、shell、PowerShell 或 curl-pipe-bash 行为。
- 代码混淆严重且没有合理解释。
- 强制遥测或上传数据，且无法关闭。
- 本地试跑前就要求生产系统写权限。
- issue 中存在未解决的安全、数据丢失、凭证泄露或恶意行为报告。

## 敏感数据规则

不要给新工具：

- 银行密码
- 支付平台密码
- 收款账号凭证
- 二次验证码
- 浏览器 cookie
- 真实浏览器配置
- 生产 API token
- 客户数据
- 医疗、金融、法律、身份等敏感数据
- 真实用户数据库

第一次测试只用：

- 假数据
- 临时账号
- 本地文件夹
- 示例 repo
- 最小权限 token
- 能只读就只读

## 10 分钟工具审查

安装前先检查：

### 1. 来源

- 项目是否在公开 repo 中？
- 维护者是否可识别？
- README 是否清楚说明工具做什么？
- 是否有示例、文档或截图？
- 最近是否有 commit 或 release？

### 2. License

- 是否有 license 文件？
- 是否允许商用？
- 是否要求署名？
- 是否有 copyleft 义务？
- 是否符合你的使用方式？

如果 license 缺失，最多只能标为 `Study only`。

### 3. 安装行为

重点检查：

- `postinstall`
- shell 脚本
- PowerShell 脚本
- 二进制下载
- 安装阶段网络请求
- native extension
- 未锁定版本的依赖

如果你无法解释安装过程做了什么，不要在主力机器上安装。

### 4. 运行权限

问自己：

- 它会读文件吗？
- 它会写文件吗？
- 它会执行命令吗？
- 它会打开浏览器吗？
- 它会访问剪贴板吗？
- 它会调用外部 API 吗？
- 它会上传代码、日志、截图或 prompt 吗？
- 它会读取环境变量吗？

权限越多，隔离要求越高。

### 5. 凭证处理

检查工具是否：

- 读取 `.env`
- 打印密钥到日志
- 把 prompt 或代码发送到外部服务
- 在本地存储 token
- 要求过宽 OAuth 权限
- 复用真实浏览器配置

如果凭证行为不清楚，标为 `Use with isolation` 或 `Reject`。

### 6. 维护状态

检查：

- 最近 commit
- 最近 release
- open issues
- 维护者回复
- 破坏性变更
- 安全相关报告

无人维护不一定代表危险，但不应在缺少额外审查的情况下进入自动化收入系统。

## 隔离等级

| 等级 | 适用情况 |
| --- | --- |
| 本地假数据测试 | 工具简单、有 license、有文档、不需要凭证。 |
| 临时文件夹 | 工具会写文件或生成代码。 |
| 临时 repo | 工具会修改项目结构或提交文件。 |
| 临时账号 | 工具要连接 GitHub、AI 平台、邮箱、存储或社交平台。 |
| 容器 / 虚拟机 | 工具会跑脚本、装大量依赖、下载二进制或行为不清楚。 |
| Reject | 工具需要支付账号、真实 cookie、生产 token 或敏感数据才能测试。 |

## 安全试跑方案

在真实工作流中使用前：

1. 创建临时文件夹。
2. 使用小型示例 repo。
3. 禁用真实密钥访问。
4. 使用假输入数据。
5. 只运行文档中的命令。
6. 记录哪些文件被修改。
7. 检查是否有网络请求。
8. 先人工审查生成结果，再复制到真实项目。
9. 发现异常就删除试跑文件夹。
10. 做出结论：Use、Use with isolation、Study only 或 Reject。

## AI 编程 Agent 专项检查

给 AI 编程 Agent 分配任务前，确认：

- 它是否只能读取必要目录？
- 它是否能避开支付、银行、税务、身份和账户动作？
- 它是否能只用假数据工作？
- 它是否不需要生产凭证？
- 它是否能说明改了哪些文件？
- 它是否能运行测试且不改无关文件？
- 你是否能先审查 diff 再接受修改？

不要允许 agent：

- 代你登录支付平台
- 转账或提现
- 提交税务或身份认证
- 使用真实客户数据且没有清晰规则
- 在未审查的情况下运行第三方安装脚本

## 简单评分

每项 0 到 2 分。

| 因素 | 0 | 1 | 2 |
| --- | --- | --- | --- |
| License | 缺失或不兼容 | 有但需审查 | 清晰可用 |
| 维护状态 | 无人维护或不清楚 | 有一些活动 | 足够活跃 |
| 文档 | 缺失 | 基础说明 | 示例清楚 |
| 安装安全 | 可疑 | 需要隔离 | 简单可解释 |
| 凭证安全 | 不安全 | 不清楚 | 最小权限 |
| 可复现性 | 难复现 | 有步骤不清楚 | 设置清楚 |
| 退出路径 | 强锁定 | 部分可导出 | 容易移除 |

建议结论：

- 12-14：Use
- 9-11：Use with isolation
- 6-8：Study only
- 0-5：Reject

硬性拒绝规则优先于分数。
